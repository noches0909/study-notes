# 微前端介绍

## 什么是微前端

微前端是把一个大型前端系统拆成多个可以独立开发、构建、发布和运行的子应用，再由一个主应用把它们组合成完整产品的架构方式。

它解决的不是“组件复用”本身，而是**多团队、多业务、多技术栈长期演进**的问题。

## 适合解决什么问题

- 巨石前端过大：构建慢、发布风险高、多人协作容易互相影响。
- 多团队独立交付：每个业务域希望拥有自己的代码仓库、技术选择和发布节奏。
- 渐进式迁移：例如从 Vue 2 迁移到 Vue 3、React，或重构老系统时不想一次性推倒重来。
- 多系统统一入口：多个后台、运营台、工作台需要合并成一个产品体验。

微前端不是银弹。如果只是一个中小型项目，或者只是想复用几个组件，优先考虑 monorepo、组件库、npm 包、路由拆分或普通懒加载。

## 核心思路

一个典型微前端系统通常包含：

- **主应用**：负责整体布局、顶层路由、登录态、权限、菜单、子应用加载和全局通信。
- **子应用**：负责各自业务页面，可以独立开发、构建、部署。
- **运行时容器**：根据路由或配置加载子应用，完成挂载、卸载、隔离和资源管理。
- **共享契约**：统一登录态、权限、设计系统、埋点、错误监控和跨应用通信规范。

常见生命周期：

```txt
匹配路由 -> 加载资源 -> 初始化 -> 挂载 -> 运行 -> 卸载
```

## 常见实现方式

### 1. iframe

iframe 是最简单、隔离最强的方案：每个子应用运行在独立浏览上下文中。

优点：

- JS、CSS、全局变量天然隔离。
- 子应用改造成本低。
- 适合嵌入外部系统、低频页面、强隔离页面。

缺点：

- 路由、弹窗、滚动、浏览器历史、登录态和通信处理麻烦。
- 性能和体验容易打折，例如首次加载、页面切换、弹窗层级。
- 与主应用深度融合时成本高。

### 2. 路由分发式框架

主应用根据路由加载不同子应用，常见框架有 `single-spa`、`qiankun`、`Garfish`。

适合场景：后台系统、运营平台、业务页面级拆分。

基本模型：

```js
import { registerApplication, start } from "single-spa"

registerApplication({
  name: "react-app",
  app: () => System.import("react-app"),
  activeWhen: ["/react"],
})

start()
```

特点：

- 以“应用”为拆分单位。
- 主应用统一控制路由和生命周期。
- 框架通常提供资源加载、沙箱、样式隔离、预加载等能力。

注意：沙箱和样式隔离不是绝对安全边界，更多是工程层面的冲突控制。

### 3. Web Components 容器

代表方案：`micro-app`、`wujie`。

它们通常通过自定义元素承载子应用：

```html
<micro-app name="user" url="https://example.com/user/"></micro-app>
```

特点：

- 接入方式接近使用 HTML 标签。
- 容器边界清晰，适合把子应用当作页面块嵌入。
- 可结合 Shadow DOM、样式改写、Proxy 沙箱或 iframe 沙箱实现隔离。

其中 `wujie` 的思路是 WebComponent 容器 + iframe 沙箱；`micro-app` 更强调类 WebComponent 的低成本接入。

### 4. Module Federation

Module Federation 是 webpack 5 引入的运行时代码共享能力，也可以通过 Vite/Rollup 插件使用。

它不是传统“页面级微前端框架”，而是让不同构建产物在运行时暴露和消费模块：

```js
new ModuleFederationPlugin({
  name: "host",
  remotes: {
    user: "user@https://example.com/user/remoteEntry.js",
  },
  shared: ["react", "react-dom"],
})
```

适合场景：

- 跨应用共享页面、组件或业务模块。
- 多团队独立发布，但希望复用运行时依赖。
- 对加载性能、依赖共享和模块粒度有更高要求。

风险点：

- 共享依赖版本需要治理，尤其是 React、Vue、状态库和设计系统。
- 远程模块失败会直接影响宿主页面，需要降级、超时和错误边界。
- 构建工具、模块格式、缓存策略要提前约定。

## 常见框架对比

| 方案 | 特点 | 适合场景 |
| --- | --- | --- |
| iframe | 原生强隔离，集成体验较弱 | 外部系统、强隔离、低频页面 |
| single-spa | 生命周期和路由编排核心 | 需要自定义能力强的团队 |
| qiankun | 基于 single-spa，提供沙箱、预加载、样式隔离等能力 | 国内中后台、页面级拆分 |
| Garfish | 路由驱动，跨框架，接入较简洁 | 多框架应用组合 |
| micro-app | 类 WebComponent 使用方式，`<micro-app>` 嵌入 | 希望低成本接入 |
| wujie | WebComponent + iframe 沙箱，隔离能力强 | 重视隔离、保活和切换体验 |
| Module Federation | 运行时模块共享，粒度更细 | 组件/模块级拆分、独立构建 |

## 需要重点治理的边界

### 路由

- 主应用分配一级路由，例如 `/user/*`、`/order/*`。
- 子应用要配置自己的 `base`、`basename` 或 `publicPath`。
- 避免主应用和子应用同时抢浏览器历史。

### 样式

- 优先使用 CSS Modules、CSS-in-JS、BEM、命名空间或设计系统约束。
- 框架提供的 scoped CSS、Shadow DOM、样式改写可以减少冲突，但不能替代规范。
- 全局样式、重置样式、UI 库样式最容易污染。

### JavaScript 沙箱

- 常见实现有 Proxy 沙箱、快照沙箱、iframe 沙箱。
- 沙箱主要防止全局变量污染，不等同于安全隔离。
- 子应用应避免随意写 `window`、污染原型、注册不清理的全局事件。

### 通信

优先让通信保持简单：

- 主应用通过 props 下发用户信息、权限、主题等只读数据。
- 子应用通过事件或约定 API 通知主应用。
- 跨应用共享复杂状态要谨慎，容易把系统重新耦合成巨石。

### 发布与监控

- 子应用独立部署，但必须有版本、灰度、回滚和资源缓存策略。
- 远程资源加载失败时要有兜底页面。
- 错误监控、性能监控、埋点字段要统一，否则问题难定位。

## 实践建议

1. 先按业务域拆，不要按技术栈拆。
2. 主应用只做壳、路由和公共能力，避免承载业务逻辑。
3. 子应用之间不要互相直接依赖，通过主应用或稳定协议通信。
4. 公共依赖不要无限共享，优先共享稳定的基础库和设计系统。
5. 为加载失败、权限失败、版本不兼容准备降级方案。
6. 小项目不要过早引入微前端，复杂度会先于收益到来。

## 总结

微前端的价值是把大型前端系统拆成可独立演进的多个应用。它带来了团队自治、独立发布和渐进迁移能力，也带来了路由、样式、依赖、通信、监控和发布治理成本。

选型时可以简单判断：

- 页面级拆分：优先看 `qiankun`、`Garfish`、`micro-app`、`wujie`。
- 模块级共享：优先看 Module Federation。
- 强隔离或外部系统嵌入：iframe 仍然有效。
- 项目规模不大：先别上微前端。
